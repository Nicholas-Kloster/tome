Three architecture approaches

All three are public CLI tools on GitHub. The difference is language, distribution, and how the O'Reilly reading connects to the output.

# Install
pip install tome-osint
# Two modes
tome scan "Ollama" # offline DB
tome research "vLLM" # reads O'Reilly live
# Pipeline integration
visor-chain-runner.sh ollama
→ step 0: tome scan "Ollama"
→ step 1: jaxen import ...
Platform profiles stored in ~/.tome/profiles.db
Research mode uses Playwright + your O'Reilly session

A — Python CLI (recommended)

Ships with a pre-built intelligence DB for ~30 AI/ML platforms. Research mode reads O'Reilly live via Playwright for new platforms. Outputs dorks, aimap probe configs, nuclei sketches. Natural fit since Playwright is already in the stack.

Pros

  • Playwright already works in your env
  • Easy to extend the DB via PRs
  • Claude API can power extraction

Cons

  • pip install vs single binary
  • Research mode requires O'Reilly sub
# Install (like aimap)
go install github.com/nuclide-research/tome@latest
# Use
tome profile ollama
tome dorks chromadb --format jaxen
tome list
# Intelligence corpus embedded in binary
Offline. No O'Reilly needed to use it.
Research mode: separate Python script
Corpus updated in repo as JSON files
Community PRs add new platform profiles

B — Go binary (like aimap)

Single-binary distribution. Intelligence corpus compiled in. Everyone uses it offline — no O'Reilly needed at runtime. Research mode (book reading) is a separate companion script. Same install pattern as aimap.

Pros

  • Zero dependencies to run
  • Matches aimap install pattern
  • Community usable without O'Reilly

Cons

  • Research mode decoupled from main binary
  • Go build complexity vs Python
# The corpus IS the product
platforms/
ollama.json ← built from 4 O'Reilly books
open-webui.json ← ch3,ch5,ch8 extracted
chromadb.json ← 2 books, 6 chapters
# CLI queries it
tome query ollama --format dorks
tome query ollama --format aimap-probe
# Each platform file cites the books
"source": "Ollama in Action, ch3, O'Reilly"

C — Corpus-first

The research output (book-derived platform profiles) is the primary artifact, version-controlled as JSON files. The CLI just queries the corpus. Novel angle: cited intelligence — every finding links back to the exact O'Reilly chapter it came from.

Pros

  • Transparent provenance
  • Git-diffable as data changes
  • Community can read the sources

Cons

  • Corpus freshness depends on maintainers
  • Less dynamic than live-read mode

All three ship on GitHub, slot into the visor chain as step 0, and output the same artifacts: default ports · API paths · Shodan dorks · aimap probe config · nuclei template skeleton
